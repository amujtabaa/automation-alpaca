# Design Systems Vault

A reference catalog of **512** public brand design systems (500 from shadcn.io, 12 from designmd.co). Each entry summarizes a brand's marketing/product design language — palette, type, geometry, motion register — as a one-paragraph brief.

Cross-source deduplicated: when a brand appears in both directories the shadcn.io entry is kept and the designmd.co duplicate dropped, so every brand appears once.

## How to use this vault

1. **Find candidates.** Search by brand or by aesthetic trait. The richest search target is `systems.json` (full descriptions, greppable):
   ```
   rg -i "near-black|monospace|brutalist|aubergine" design-systems/systems.json
   rg -i '"name": "Stripe"' design-systems/systems.json
   ```
   Or skim this file's source sections below to browse by name.
2. **Read the brief.** Pull the matching object from `systems.json` for the full one-paragraph description (palette, type, geometry, motion).
3. **Fetch the live spec when committing to a system.** The brief is a teaser; the full DESIGN.md-style spec lives at the entry's `url`. When a build session settles on a system, fetch that URL (WebFetch, or the `dev-browser` skill if the host rate-limits) for the complete component-level reference, then implement in the chosen direction.

> These are *references to learn from*, not drop-in themes. Extract the principles (the "why" behind a palette or type choice) and execute them in a cohesive, context-specific build — do not clone a brand pixel-for-pixel.

## Sources

- **shadcn.io** — `https://www.shadcn.io/design/{slug}` — each links to a full DESIGN.md spec with component counts.
- **designmd.co** — `https://www.designmd.co/d/{slug}` — pairs with the designmd "taste skill" (mined into `../taste-engineering.md`).
