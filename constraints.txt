# constraints.txt — pinned dependency lock for reproducible builds.
#
# WHY: requirements.txt intentionally stays abstract (`>=`), which let a fresh CI
# runner silently drift to mypy 2.x + numpy 2.5 (mypy then rejected numpy's PEP-695
# stubs and the `mypy app/` gate went red before pytest even ran) and to newer
# Hypothesis. This file pins the FULL resolved dependency closure so CI and local
# installs are byte-for-byte reproducible and a transitive bump can't break the
# gate out from under a green PR.
#
# HOW IT IS USED: CI (and you, locally) install with
#     pip install -r requirements.txt -c constraints.txt
# requirements.txt still declares WHAT we depend on; this file freezes WHICH
# versions. A constraint only applies to a package that actually gets installed,
# so this is safe across the 3.11 / 3.12 CI matrix.
#
# REGENERATE (after an intentional dependency change):
#     python -m pip install -r requirements.txt --dry-run --ignore-installed \
#         --report report.json
#     # then emit "<name>==<version>" for every entry in report.json["install"]
# and let the 3.11 + 3.12 CI matrix validate the new pins.
#
# NOTE: Hypothesis is pinned for REPRODUCIBILITY only. (Historical: an earlier
# flaky TestSqliteLifecycle failure exposed a real INV-034/INV-036 tension on the
# flatten surface — since RESOLVED: carve-out decided 2026-07-09, WO-0015 shipped
# the operator-visible deferral, REV-0003 cleared the review gate, and the
# lifecycle test rule was reconciled. See
