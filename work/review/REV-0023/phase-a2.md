## 5. Independent implementer-seat verification of the P0 (completeness-0)

Not taken on the finders' word — re-traced against the code at tip before pinning
(the WO-0031 "confirmed can still be vacuous" lesson):

- **(A) envelope-backed intent stays APPROVED.** The only writer of
  `SellIntentStatus.ORDERED` is `create_order_for_sell_intent` (memory.py:1487) — the
  LEGACY single-order handoff. `approve_envelope_activation` (memory.py:1287-1362) and
  every other envelope op never touch `_sell_intents`. CONFIRMED.
- **(B) single-ACTIVE is per-intent, not per-symbol.** `_other_active_envelope_unlocked`
  (memory.py:916-929) filters on `sell_intent_id`; the sqlite twin is a partial unique
  index `ON(sell_intent_id) WHERE status='active'`. No symbol-scoped exclusivity exists
  anywhere. CONFIRMED.
- **(C) close expires the intent, orphans the envelope.** `_close_session_unlocked`
  (memory.py:2919-2925) EXPIREs every PENDING/APPROVED sell_intent **whose
  `session_id == the closing session's id`**, and reads/writes NO envelope. CONFIRMED —
  with a precondition I found while validating the pin (see reachability).

**Mechanism reproduced on BOTH stores (pinned), but reachability is narrower than the
finder stated — recorded honestly.** The double-mandate reproduces on memory AND sqlite
(`tests/test_rev0023_phase_a2_pins.py`, 2 ACTIVE envelopes) **only when the backing intent
is APPROVED-not-ORDERED AND carries the closing session's `session_id`**. Two facts narrow
the production trigger:
1. `create_sell_intent` passes `session_id=<arg>` (default **None**) straight through — it
   does NOT auto-stamp the current session. An intent created without an explicit
   `session_id` has `session_id=None`, so `_close_session_unlocked`'s `== session.id`
   filter never matches it and close never expires it (dedup then holds, env2 is refused —
   I verified this: the unstamped repro does NOT reproduce).
2. The two *automatic* intent creators that DO stamp `session_id`
   (`open_protection_exit`, `flatten_position`) each **dispatch a legacy order**, moving
   the intent to **ORDERED** — which close does not expire. So no current automatic path
   produces the APPROVED-not-ORDERED, session-stamped, envelope-backed intent the defect
   needs.

**Net:** this is a real **store-contract-level single-mandate (INV-031/032 / ADR-010)
violation** — the API *permits* two ACTIVE envelopes for one symbol — but it is **NOT an
active oversell in today's shipped wiring**. It goes live the moment the envelope-native
exit flow (create sell_intent → approve an envelope for it, no legacy order dispatch, with
a session-stamped intent) is wired, which is the natural next integration step. Verdict:
**P0 at the contract level; MUST-FIX before the envelope-native creation flow is wired /
before T5 merge relies on the single-mandate guarantee.** Pinned strict-xfail on both
stores.
